สรุปความรู้เรื่อง IWS จากแชทนี้

รวบรวมเฉพาะส่วนที่เกี่ยวกับ IWS โดยตรง เอาไปวางในแชทใหม่สำหรับโปรเจกต์ "จำลอง IWS บน PC" ได้เลย

1. IWS คืออะไร

Integrated Web Services for i — ฟีเจอร์ของ IBM i ที่เปิด RPG/COBOL program หรือ SQL statement ให้กลายเป็น REST/SOAP API โดยไม่ต้องเขียน middleware เพิ่ม ทำงานบน WebSphere Application Server (WAS Base) ที่ built-in มากับ IBM i

RPG Program / SQL
→
IWS (deploy ผ่าน IWS Studio)
→
REST URL
→
เว็บเรียกผ่าน fetch()

2. วิธี Deploy 2 แบบ

แบบใช้เมื่อตัวอย่าง
SQL statementมีแค่ query ธรรมดา ไม่มี business logic ซับซ้อนSELECT * FROM TABLE WHERE ID = ?
Programมี RPG/COBOL service program (*SRVPGM) ที่มี logic อยู่แล้วเลือก library + object + procedure แล้ว map parameter
สำคัญ: IWS ทำหน้าที่ "mapping" ชื่อ parameter/field ระหว่างของจริงกับที่ฝั่งเว็บเห็นด้วย — ไม่ใช่แค่ห่อ URL เฉยๆ (ดูตัวอย่างจริงข้อ 4)

3. วิธีเข้าถึง IWS Studio / Admin

เครื่องมือURL ทั่วไปใช้ทำอะไร
IBM Navigator for ihttps://<server>:2001เมนู IBM i Management → Integrated Web Services
IBM Web Administration for ihttps://<server>:2010/HTTPAdminWeb Services Server instance, deploy service ใหม่
IWS Studio (บางระบบ)https://<server>:2005Port แยกเฉพาะบางเวอร์ชัน
ต้องมีสิทธิ์: user profile ระดับ authority สูง (ไม่ใช่ authority สูงสุด แต่ต้องเข้าถึง HTTP Admin ได้) — บัญชีทั่วไป (รวมถึง pub400 สาธารณะ) มักไม่ได้สิทธิ์นี้ ได้แค่ 5250 + database access เท่านั้น

4. ตัวอย่าง Endpoint จริงที่เจอมา (ใช้อ้างอิงได้)

Endpoint: scanner/minpop

GET http://as400webappdev/api/web/serveices/scanner/minpop?empid=25066

Response:
{
  "P_EMPNAME": "I.KIDAKORN",
  "P_ISFOUND": "1"
}

Field จริงในตาราง/RPG: CDOPCD (Operator Code, key) / NMOPCD (Operator Name)

IWS mapping: CDOPCD → empid (parameter ที่รับ) และ NMOPCD → P_EMPNAME (field ที่ตอบกลับ)

Endpoint ที่สอง: talinf

GET .../talinf?lotno=05245-0001-421-001

Format ของ lotno: XXXXX-XXXX-XXX-XXX
Response fields: "ASSEMBLY LOT NO.", "DEVICE CODE", "DEVICE NAME", 
                 "PROCESS CODE", "QTY"
Pattern ร่วมของทั้งสอง: ทุก endpoint คืนค่า P_ISFOUND เป็น string "1"/"0" บอกว่าเจอข้อมูลไหม — ควรทำตามรูปแบบเดียวกันตอนจำลอง

5. เรื่อง Reverse Proxy Path

ในการใช้งานจริง เว็บไม่ได้เรียก IWS server ตรงๆ (IP:port) แต่เรียกผ่าน path บนโดเมนเดียวกัน (เช่น /api/web/serveices/...) ซึ่งมี URL Rewrite rule ใน web.config (IIS) คอยแปลง path นี้ไปหา IWS server จริงที่อยู่หลังบ้านให้ — ทำให้เป็น same-origin และไม่ติดปัญหา CORS

6. สถาปัตยกรรมจำลอง IWS ที่ร่างไว้แล้ว (ODBC + Node.js)

เนื่องจากไม่มีสิทธิ์ deploy IWS จริงได้ (ทั้งที่บริษัทและ pub400) แนวทางคือแทนที่ IWS ด้วยการต่อฐานข้อมูลตรงผ่าน ODBC แล้วเขียน REST API เอง:

Frontend (fetch)
→
Node.js + Express (จำลอง IWS)
→
ODBC Driver
→
DB2 for i (pub400 หรือเครื่องอื่น)
องค์ประกอบ IWSตัวจำลองบน PC
IWS Studio (deploy service)เขียนไฟล์ routes/xxx.js เอง
Parameter/Field mappingเขียนเองใน res.json({...}) ก่อนตอบกลับ
REST endpoint ที่ได้Express.js route (app.get('/api/minpop', ...))
Connection to DB2odbc npm package + IBM i Access Client Solutions driver

โครงสร้างโปรเจกต์ที่แนะนำ (ขยาย endpoint ใหม่ได้ง่าย):

my-iws-api/
  ├── .env              (DB_DSN, DB_USER, DB_PASSWORD)
  ├── server.js          ← ผูก route ทั้งหมดเข้าด้วยกัน
  ├── db.js              ← ฟังก์ชันเชื่อมต่อ ODBC ใช้ร่วมกันทุก endpoint
  └── routes/
        ├── minpop.js
        └── talinf.js
ข้อจำกัดสำคัญ: Node.js API นี้ต้องรันอยู่บนเครื่อง PC ตลอดเวลาที่ใช้งาน (ปิดเครื่อง = endpoint หายไปด้วย) — เหมาะกับ "ฝึกส่วนตัว/จำลอง" เท่านั้น ไม่ใช่ระบบ production จริง

7. ข้อควรระวัง

  • ห้ามใช้ข้อมูลจริงของบริษัทไปทดสอบบน pub400 หรือระบบสาธารณะใดๆ ใช้ข้อมูลสมมติเท่านั้น
  • ใช้ parameterized query เสมอ ป้องกัน SQL Injection (ห้ามต่อ string ตรงๆ ใน query)
  • ห้าม commit ไฟล์ .env ที่มี password ขึ้น GitHub
