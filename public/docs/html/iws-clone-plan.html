Project Plan · IBM i / AS400

สร้าง GUI ของตัวเอง แทน IWS

สรุปแนวทางพัฒนา middleware/API server ที่รันบน PC ของคุณเอง ทำหน้าที่แทน Integrated Web Services (IWS) ที่แอดมินเคยตั้งไว้ให้ — เพื่อให้พัฒนาเว็บเชื่อม AS400 ได้เอง โดยไม่ต้องรอใคร

Pattern ที่ต้องทำตาม: ทุก endpoint ของ IWS จริงคืนค่า P_ISFOUND เป็น string "1"/"0" บอกว่าเจอข้อมูลไหม — ตอนจำลองควรตอบกลับ รูปแบบเดียวกัน เพื่อให้ frontend เดิม (ถ้าย้ายมาใช้ API ใหม่) ไม่ต้องแก้โค้ดมาก และการ mapping ชื่อ field เป็นสิ่งที่ต้องทำเองใน res.json({...}) ไม่ใช่แค่ห่อ URL เฉยๆ
IWS จริงรันบน WebSphere (WAS Base) built-in บน IBM i Deploy ผ่าน IBM Navigator :2001 หรือ Web Admin :2010/HTTPAdmin ต้อง authority สูง — pub400/user ทั่วไปไม่มีสิทธิ์นี้ Production จริงเรียกผ่าน reverse proxy (IIS URL Rewrite) ไม่ใช่ IP:port ตรง

04 · อยาก ได้ GUI บน Windows

ห่อเป็นแอป Desktop ด้วย Electron

รัน Express server ไว้ข้างในตัวแอปเอง แล้วแสดงหน้าจัดการ endpoint เป็นหน้าต่าง Windows ปกติ — กด .exe เดียวเปิดได้เลย ไม่ต้องเปิด browser แยก

แนวคิด: Electron = Chromium + Node.js ในตัวเดียวกัน จึงเรียก itoolkit/odbc จากฝั่ง main process ได้ตรงๆ แล้วส่งผลลัพธ์ขึ้นหน้า GUI ผ่าน IPC — ได้ทั้ง "API server" และ "หน้าต่างโปรแกรม" ในไฟล์เดียว
05 · เริ่มลงมือ

ขั้นตอนแรก (เชื่อม pub400)

  1. ติดตั้ง IBM i Access ODBC Driver บนเครื่อง PC ของคุณ
  2. สร้างโปรเจกต์ — npm install odbc express dotenv
  3. ทดสอบเชื่อมต่อ ผ่าน SYSTEM=www.pub400.com;PORT=8471
  4. เช็ค XMLSERVICE บน pub400 ว่า user ทั่วไปเรียกได้ไหม (สำหรับเรียก RPG/CL)
  5. ทำ endpoint แรก เช่น query ตาราง QIWS.QCUSTCDT ก่อน
  6. ขยายทีละ endpoint จนครอบคลุมสิ่งที่ IWS เดิมทำอยู่
06 · Roadmap

แผนพัฒนา 4 เฟส

เฟสเป้าหมายสิ่งที่ต้องทำ
Phase 1
Connectivity		 เชื่อม PC ↔ pub400 ได้จริง ติดตั้ง ODBC driver, ต่อ odbc, query DB2 อ่านอย่างเดียวได้สำเร็จ
Phase 2
Program Call		 เรียก RPG/CL ได้แบบ IWS เดิม ตั้งค่า/ตรวจสอบ XMLSERVICE, ติดตั้ง itoolkit, ทำ endpoint เรียกโปรแกรมตัวอย่าง
Phase 3
API Layer		 มี API มาตรฐานใช้ซ้ำได้ ออกแบบ route ตาม resource, เพิ่ม validation, error handling, connection pooling
Phase 4
Desktop GUI		 แอป Windows ใช้งานจริง ห่อด้วย Electron, ทำหน้าจัดการ/ทดสอบ endpoint, เก็บ config เป็น .env หรือ settings UI
07 · Prompt สำหรับ Agent

สั่งทีละ Phase (คัดลอกไปวางได้เลย)

ใช้ทีละ prompt ตามลำดับ เฟสไหนรันผ่านแล้วค่อยเอา prompt เฟสถัดไปไปสั่งต่อ ไม่ต้องรวมกันในครั้งเดียว

Phase 1 — Connectivity
ก่อนเริ่มทำอะไรทั้งหมด อ่านไฟล์เหล่านี้ก่อน: - CLAUDE.md (บริบทโปรเจกต์และกฎตายตัว) - .ai/PLAN.md (สถานะงานปัจจุบัน) - .ai/skills/ibmi-pub400/SKILL.md (การเชื่อมต่อ pub400 และ pattern การตอบกลับ) จากนั้นสร้างโปรเจกต์ Node.js ชื่อ "iws-clone" ทำทีละขั้นตอนดังนี้: 1. รัน npm init -y แล้วติดตั้ง express, odbc, dotenv 2. สร้างไฟล์ .env.example ที่มี PUB400_USER= และ PUB400_PASS= (ไม่ใส่ค่าจริง) 3. สร้าง db.js: ทำ connection pool ด้วย odbc.pool() เชื่อมต่อ DRIVER={IBM i Access ODBC Driver};SYSTEM=www.pub400.com;PORT=8471; UID=<จาก .env>;PWD=<จาก .env> 4. สร้าง server.js เป็น Express บนพอร์ต 3000 พร้อม endpoint เดียวคือ GET /api/health ที่เช็คว่าเชื่อม pub400 ได้ไหม คืน { status: "ok" } หรือ { status: "error", message } 5. เพิ่ม GET /api/customers query ตาราง QIWS.QCUSTCDT คืนเป็น JSON โดยตอบกลับ ตาม pattern ของ IWS จริง คือมี field P_ISFOUND เป็น string "1"/"0" บอกว่า เจอข้อมูลไหม เช่น { "P_ISFOUND": "1", "data": [...] } หรือถ้าไม่เจอ { "P_ISFOUND": "0", "data": [] } 6. ทำตัวอย่าง field mapping ให้เห็นชัดเจน 1 endpoint เช่น GET /api/employee?empid=xxx ที่ map ชื่อ field จริงในตาราง (เช่น รหัส พนักงาน) ให้ออกมาเป็นชื่อ field ที่ต้องการฝั่ง response เอง ไม่ใช่คืนชื่อ column ดิบจากฐานข้อมูลตรงๆ 7. เขียน README.md อธิบายการติดตั้ง IBM i Access ODBC Driver บน Windows, วิธีตั้งค่า .env, วิธีรัน, และอธิบาย pattern P_ISFOUND ที่ใช้ทุก endpoint 8. ทดสอบรันจริงแล้วรายงานผลกลับมาว่า endpoint ไหนทำงานได้บ้าง ห้ามใช้ itoolkit ในเฟสนี้ โฟกัสแค่ ODBC ให้เสถียรก่อน ห้ามใช้ข้อมูลจริงของบริษัทใดๆ ทดสอบบน pub400 ใช้ข้อมูลสมมติ/ตัวอย่างเท่านั้น ทุก query ต้องใช้ parameterized query ห้ามต่อ string ตรงๆ เข้า SQL
Phase 2 — Program Call (RPG/CL)
ก่อนเริ่มทำอะไรทั้งหมด อ่าน CLAUDE.md, .ai/PLAN.md และ .ai/skills/ibmi-pub400/SKILL.md ก่อน เพื่อดูว่า Phase 1 ทำอะไรไปแล้วบ้าง จากนั้นต่อยอดจากโปรเจกต์ "iws-clone" เดิม ทำทีละขั้นตอนดังนี้: 1. เช็คก่อนว่า pub400 เปิดใช้ XMLSERVICE ให้ user ทั่วไปหรือไม่ ถ้าไม่แน่ใจ ให้เขียนสคริปต์ทดสอบสั้นๆ เพื่อยืนยันก่อนเขียนโค้ดจริง 2. ติดตั้ง itoolkit และ fast-xml-parser 3. สร้าง ibmi.js: ฟังก์ชัน callProgram(programName, params) ที่ห่อ itoolkit Connection + ProgramCall ไว้ใช้ซ้ำได้ทุก endpoint 4. เพิ่ม endpoint ตัวอย่าง POST /api/program/:name รับพารามิเตอร์จาก body แล้วเรียกโปรแกรมผ่าน callProgram() คืนผลลัพธ์เป็น JSON 5. เพิ่ม endpoint สำหรับรันคำสั่ง CL เช่น POST /api/command รับ command string แล้วรันผ่าน CommandCall ของ itoolkit 6. เพิ่ม error handling เฉพาะของ itoolkit (เช่น timeout, XML parse error) ให้คืน error message ที่อ่านง่าย ไม่ใช่ raw XML 7. อัปเดต README.md เพิ่มส่วนอธิบายวิธีเรียกโปรแกรมผ่าน endpoint ใหม่ 8. ทดสอบเรียกโปรแกรมง่ายๆ 1 ตัวจริงบน pub400 แล้วรายงานผล
Phase 3 — API Layer
ก่อนเริ่มทำอะไรทั้งหมด อ่าน CLAUDE.md และ .ai/PLAN.md ก่อน เพื่อดูว่า Phase 1-2 ทำอะไรไปแล้วบ้าง จากนั้นปรับโครงสร้างโปรเจกต์ "iws-clone" ให้เป็น API ที่ใช้งานจริงได้ ทำทีละขั้นตอน: 1. จัดกลุ่ม routes ใหม่ตาม resource แยกเป็นไฟล์ใน /routes (เช่น routes/customers.js, routes/programs.js, routes/commands.js) 2. เพิ่ม validation ด้วย library เช่น zod ตรวจสอบ input ทุก endpoint ก่อนส่งไปหา pub400 3. จำกัดสิทธิ์ /api/query ให้รับเฉพาะ SELECT statement เท่านั้น ปฏิเสธ INSERT/UPDATE/DELETE/DROP ด้วย regex หรือ SQL parser 4. เพิ่ม logging กลาง (เช่น morgan หรือ pino) log ทุก request/response พร้อมเวลาและ endpoint ที่เรียก 5. เพิ่ม rate limiting เบื้องต้นกันเรียกถี่เกินไปจนโดน pub400 บล็อก 6. เขียน integration test อย่างน้อย 1 ไฟล์ทดสอบ endpoint หลักด้วย supertest 7. อัปเดต README.md ให้เป็นเอกสาร API แบบสมบูรณ์ (list endpoint ทั้งหมด พร้อมตัวอย่าง request/response)
Phase 4 — Desktop GUI (Electron)
ก่อนเริ่มทำอะไรทั้งหมด อ่าน CLAUDE.md และ .ai/PLAN.md ก่อน เพื่อดูว่า Phase 1-3 ทำอะไรไปแล้วบ้าง จากนั้นห่อโปรเจกต์ "iws-clone" ให้เป็นแอป Windows ด้วย Electron ทำทีละขั้นตอน: 1. ติดตั้ง electron และ electron-builder ในโปรเจกต์ 2. สร้าง main.js: ให้ Electron รัน Express server (จาก Phase 1-3) เป็น background process ตอนเปิดแอป แล้วเปิดหน้าต่างโหลดหน้า GUI 3. สร้างหน้า GUI เดียว (index.html + renderer.js) ที่มี: - ฟอร์มกรอก user/password สำหรับ pub400 (เก็บด้วย electron-store ไม่ hardcode) - ปุ่มทดสอบการเชื่อมต่อ (เรียก /api/health) - ตารางแสดงผลลัพธ์เวลาทดสอบ endpoint ต่างๆ - ช่องให้พิมพ์ SQL หรือชื่อโปรแกรมเพื่อทดสอบยิง request ได้จาก GUI เลย 4. ใช้ IPC (ipcMain/ipcRenderer) เชื่อม renderer กับ main process แทนการ เรียก HTTP ตรงจากหน้าเว็บ เพื่อความปลอดภัย (contextIsolation: true) 5. ตั้งค่า electron-builder ให้ build ออกมาเป็นไฟล์ .exe สำหรับ Windows 6. ทดสอบ build แล้วรันไฟล์ .exe จริง 1 ครั้ง รายงานผลว่าเปิดและเชื่อม pub400 ได้หรือไม่
08 · ข้อควรระวัง

Safety checklist

  1. ห้ามใช้ข้อมูลจริงของบริษัท ไปทดสอบบน pub400 หรือระบบสาธารณะใดๆ — ใช้ข้อมูลสมมติเท่านั้น
  2. Parameterized query เสมอ ป้องกัน SQL Injection ห้ามต่อ string ตรงๆ เข้า query
  3. ห้าม commit ไฟล์ .env ที่มี password ขึ้น GitHub — ใส่ใน .gitignore ตั้งแต่ commit แรก
  4. โปรเจกต์นี้เหมาะกับฝึกส่วนตัว/จำลองเท่านั้น ไม่ใช่ระบบ production — ต้องรัน Node.js อยู่บน PC ตลอดเวลาที่ใช้งาน ปิดเครื่อง endpoint ก็หายไปด้วย

--- name: ibmi-pub400 description: ใช้เมื่อต้องเขียน endpoint ที่เชื่อมต่อ IBM i / pub400, จำลอง พฤติกรรมของ IWS, หรือแปลง RPG/DB2 field ให้เป็น REST response --- # การเชื่อมต่อ pub400 SYSTEM=www.pub400.com, PORT=8471 (ไม่เข้ารหัส) หรือ 9471 (SSL) ใช้ driver "IBM i Access ODBC Driver", credential เก็บใน .env เท่านั้น # Pattern การตอบกลับแบบ IWS ทุก endpoint ต้องมี field P_ISFOUND เป็น string "1"/"0" ตัวอย่างจริง: GET /minpop?empid=25066 → { "P_EMPNAME": "I.KIDAKORN", "P_ISFOUND": "1" } # Field mapping ชื่อ column จริงในตาราง (เช่น CDOPCD, NMOPCD) ไม่ตรงกับชื่อ response (เช่น empid, P_EMPNAME) — ต้อง map เองใน res.json() เสมอ ห้ามคืนชื่อ column ดิบตรงๆ # ข้อจำกัดของ pub400 - user ทั่วไปมักไม่มีสิทธิ์ deploy IWS จริง (ต้อง authority สูง) - ไม่แน่ใจว่า XMLSERVICE เปิดให้ใช้ไหม ต้องทดสอบก่อนเขียนโค้ด Phase 2 - ห้ามใช้ข้อมูลจริงของบริษัทใดๆ ทดสอบบน pub400
ทำไมต้องแยกเป็น 4 ไฟล์: CLAUDE.md/AGENTS.md บอก "บริบทของทั้งโปรเจกต์" ให้ agent เห็นทันทีตอนเริ่มงาน, PLAN.md ติดตาม "ตอนนี้ทำถึงไหนแล้ว" จะได้สั่งงานต่อได้โดยไม่ต้องอธิบายซ้ำ, ส่วน SKILL.md เก็บ "ความรู้เฉพาะโดเมน" (pub400 + IWS pattern) แยกไว้ต่างหาก เผื่อเอาไปใช้ซ้ำกับโปรเจกต์อื่นที่เกี่ยวกับ IBM i ได้เลย
