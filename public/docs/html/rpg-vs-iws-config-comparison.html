RPG Program vs IWS Config

สองส่วนที่มักสับสนกันบ่อยที่สุดเวลาทำงานกับ AS400 — คู่มือนี้แยกให้เห็นชัดว่าอะไรเป็นของใคร และต้องติดต่อใครเวลาเจอปัญหา

🔧 RPG Program

"เครื่องยนต์" — ตรรกะทางธุรกิจจริง

  • ภาษาโปรแกรมที่รันบน IBM i (คล้ายเขียน function/class)
  • ทำงานจริง เช่น อ่าน/เขียนตาราง, คำนวณ, ตรวจเงื่อนไข
  • เก็บเป็น object บน IBM i (*PGM/*SRVPGM) ใน library
  • แก้ได้เฉพาะทีม RPG/programmer เท่านั้น
  • แก้แล้วต้อง compile ใหม่เสมอ
  • ถ้าไม่มี IWS ครอบ เรียกได้แค่จาก green screen (5250)

🌐 IWS Config

"ป้ายทางเข้า" — ตัวห่อให้เว็บเรียกได้

  • ไม่ใช่โค้ด แต่เป็นการตั้งค่า/mapping
  • บอกว่า "URL นี้ ให้ไปเรียก RPG program ตัวนี้"
  • ตั้งค่าผ่าน IWS Studio ไม่ต้องเขียนโค้ดใหม่
  • ผลลัพธ์คือ URL เช่น /api/web/services/SCANNER/MINPOP
  • แก้แล้วแค่ deploy ใหม่ ไม่ต้อง compile โปรแกรม
  • ฝั่งเว็บ (คุณ) จะยุ่งกับส่วนนี้เป็นหลัก
เปรียบง่ายๆ: RPG program คือ เครื่องยนต์ (ทำงานจริง) ส่วน IWS config คือ ป้ายทางเข้า + คู่มือใช้งาน (บอกว่าต้องเข้าทางไหน ส่งอะไรเข้าไป)

เช็คจากสิ่งที่เจอจริง — เป็นของฝั่งไหน

สิ่งที่เจอฝั่งที่เกี่ยวข้อง
ชื่อ table, field, business logic (เช่น "ถ้า status=1 ให้ทำแบบนี้")RPG Program
URL /api/web/services/..., path, HTTP method (GET/POST)IWS Config
Error 500 เพราะ SQL ผิดหรือคำนวณผิดRPG Program
Error 404 / 405 เพราะ path ผิดหรือยังไม่ deployIWS Config
แก้แล้วต้อง "compile" โปรแกรมใหม่RPG Program
แก้แล้วแค่ "deploy ใหม่" ใน IWS StudioIWS Config

ติดปัญหา ต้องถามใคร

อาการติดต่อ
อยากได้ field เพิ่ม / เปลี่ยนตรรกะการคำนวณทีม RPG
อยาก endpoint ใหม่ / เปลี่ยน path / เปลี่ยน methodแอดมิน (IWS Studio)
เรียก API แล้ว 500ทีม RPG (เช็ก joblog)
เรียก API แล้ว 404 / 405แอดมิน (เช็ก deploy/path)

สรุป: ฝั่งเว็บ (คุณ) ยุ่งกับ IWS config เป็นหลักผ่านการคุยกับแอดมิน ไม่ต้องแตะ RPG เลย ถ้าอยากเปลี่ยนตรรกะจริงๆ ต้องขอทีม RPG แก้โปรแกรมแล้ว deploy IWS ใหม่อีกที — สองขั้นตอนนี้แยกกันคนละทีมเสมอ
