webas400app / kidakorn — Dev Roadmap

IWS เทียบกับ pub400 + PC — ใช้อะไรตรงไหน

แบ่งงานให้ชัดว่า "ฝึกฝั่ง backend (IWS/RPG)" ทำที่ pub400 กับ "พัฒนาฝั่ง frontend" ทำที่ PC ตัวเอง แล้วมาต่อกันตอนไหน

สองสภาพแวดล้อม แยกหน้าที่กันชัดเจน

🌐 pub400

IBM i จริง — ฝึกฝั่ง IWS/backend

  • สร้าง RPG program / SQL table ทดลองเอง
  • ฝึก deploy service ผ่าน IWS Studio ด้วยตัวเอง (ไม่ต้องรอสิทธิ์จากแอดมันบริษัท)
  • ทดสอบ endpoint จริงผ่าน URL แบบเดียวกับที่ทำกับ SCANNER/minpop
  • ใช้ข้อมูลสมมติเท่านั้น ห้ามใช้ข้อมูลบริษัท
⇄

💻 PC ตัวเอง

พัฒนาฝั่ง frontend — Next.js/React

  • เขียน UI, component, logic ทั้งหมด
  • npm run dev ทดสอบด้วย mock data ก่อน
  • สลับไปเรียก endpoint จริงจาก pub400 หรือบริษัทได้ (แค่เปลี่ยน BASE_URL)
  • npm run build ได้ static files พร้อม deploy
⚠️ pub400 อยู่บน internet สาธารณะ ส่วน endpoint บริษัท (as400webappdev) อยู่ใน network ภายในเท่านั้น — โค้ดฝั่ง frontend ที่ทดสอบกับ pub400 จะต้องแก้ BASE_URL เป็น URL เต็มของ pub400 (ไม่ใช่ relative path แบบที่ใช้ในบริษัท เพราะคนละ origin กัน จะติด CORS ด้วย ต้องเช็ก config CORS ฝั่ง pub400 ก่อน)

เทียบ capability ของ IWS กับที่ที่ควรฝึก

Roadmap รวม (เรียงตามลำดับ)

1

สมัคร pub400 + ฝึกพื้นฐาน

สมัครบัญชี pub400.com ฟรี ลองสร้างตารางง่ายๆ (เช่น mock employee table) ด้วย SQL

pub400
2

Deploy IWS service แรกด้วยตัวเอง

ลอง deploy SQL statement เป็น REST endpoint ผ่าน IWS Studio บน pub400 — ฝึกขั้นตอนที่ปกติต้องขอแอดมินบริษัททำให้

pub400
3

สร้างโปรเจกต์ Next.js บน PC เชื่อม pub400

ใช้ pattern เดียวกับที่ทำกับบริษัท (config.ts, lib/api/, hooks) แต่ชี้ BASE_URL ไปที่ pub400 แทน — ฝึกทั้ง flow ครบวงจรโดยไม่กระทบระบบจริง

PCpub400
4

ลองฟีเจอร์ใหม่ที่ยังไม่เคยทำ

เช่น POST/PUT/DELETE (แก้ข้อมูล ไม่ใช่แค่ดึงมาดู), auth แบบง่าย, endpoint ที่ return array หลาย record — ลองบน pub400 ก่อนเสมอ

pub400
5

ย้าย pattern ที่ฝึกแล้วมาใช้กับบริษัทจริง

พอมั่นใจใน pattern แล้ว เอาโครงสร้างเดียวกันไปเสนอ/สร้าง spec ขอแอดมิน deploy endpoint จริงแบบเดียวกัน
