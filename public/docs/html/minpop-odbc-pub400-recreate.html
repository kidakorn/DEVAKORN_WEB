สร้าง endpoint แบบ minpop เดิม ด้วย ODBC

ทำ endpoint หน้าตาเดียวกับ SCANNER/minpop ที่เคยทำกับบริษัท (ใส่ empid → ได้ P_EMPNAME, P_ISFOUND กลับมา) แต่คราวนี้สร้างเองทั้งหมดผ่าน pub400

เทียบขั้นตอนแบบคู่กัน — เหมือนกันมากกว่าที่คิด

ขั้นตอนแบบ IWS (บริษัท)แบบ ODBC (pub400)
1. มีตาราง/ข้อมูลอยู่แล้วมีอยู่แล้วบน AS400 บริษัทต้องสร้างเองบน pub400 (ขั้นตอนที่ 1-2)
2. เขียน query ดึงข้อมูลแอดมิน/RPG เขียนให้คุณเขียนเอง เป็น SQL ธรรมดา
3. ทำให้เรียกผ่าน URL ได้IWS Studio ห่อให้อัตโนมัติExpress.js เขียน route เอง
4. แปลงชื่อ field จริง → ชื่อที่เว็บใช้ตั้งค่า mapping ใน IWS Studio (CDOPCD→empid)เขียนเองใน res.json({...})
5. รับ parameter (empid)?empid=25066?empid=25066 — เหมือนกันเป๊ะ
6. ตอบกลับเป็น JSON{ P_EMPNAME, P_ISFOUND }{ P_EMPNAME, P_ISFOUND } — ทำให้หน้าตาเหมือนเดิมได้เลย
7. ฝั่งเว็บเรียกใช้fetch()fetch() — โค้ดเดิมไม่ต้องแก้เลย
สรุปสั้นๆ: ขั้นตอนที่ 5-7 ฝั่งเว็บเหมือนกัน 100% ต่างกันแค่ขั้นตอน 1-4 ที่คุณต้องทำเองแทนแอดมิน (รวมถึงงาน "แปลงชื่อ field" ที่ IWS เคยทำให้อัตโนมัติ)
1
สร้างตารางบน pub400 (แทนตารางจริงที่บริษัท)

เข้า pub400 ผ่าน 5250 หรือ ACS Run SQL Scripts แล้วรัน (ใช้ชื่อ field แบบเดียวกับ AS400 จริง CDOPCD/NMOPCD แทนชื่อสมมติ):

CREATE TABLE MYLIB.OPERATOR (
  CDOPCD   CHAR(6)     NOT NULL,   -- Operator Code (key) — ชื่อ field จริงในตาราง
  NMOPCD   VARCHAR(50),            -- Operator Name — ชื่อ field จริงในตาราง
  PRIMARY KEY (CDOPCD)
);
เทียบกับของบริษัท: ตารางนี้แทนตารางจริงที่ RPG program MINPOP ไปดึงข้อมูลมา — ใช้ชื่อ field จริง (CDOPCD, NMOPCD) เพราะฝั่ง IWS ก็ทำหน้าที่ "แปลงชื่อ" ให้กลายเป็น empid/P_EMPNAME ก่อนส่งถึงเว็บอยู่แล้ว เราจะจำลองขั้นตอนแปลงชื่อนี้เองในขั้นตอนที่ 4
2
ใส่ข้อมูลตัวอย่าง (ห้ามใช้ข้อมูลจริงบริษัท)
INSERT INTO MYLIB.OPERATOR (CDOPCD, NMOPCD) VALUES
  ('25066', 'I.KIDAKORN'),
  ('25067', 'S.SOMCHAI'),
  ('25068', 'A.ANAN');
3
ทดสอบ SQL ก่อนเขียนโค้ด (สำคัญมาก — เช็คให้ผ่านก่อนเสมอ)
SELECT CDOPCD, NMOPCD FROM MYLIB.OPERATOR WHERE CDOPCD = '25066';

ต้องได้แถวข้อมูลกลับมา 1 แถวก่อน ถึงจะไปขั้นต่อไป — เหมือนตอนที่เคยทดสอบ URL SCANNER/minpop?empid=25066 ตรงๆ ใน browser ก่อนต่อ IWS

4
เขียน Node.js API — โครงสร้างที่ขยาย endpoint เพิ่มได้ง่าย

ตั้งชื่อโปรเจกต์แบบกลางๆ ไม่ผูกกับ endpoint เดียว เพราะจะมี talinf หรือตัวอื่นตามมาทีหลัง:

mkdir my-iws-api
cd my-iws-api
npm init -y
npm install express odbc cors dotenv
// .env
DB_DSN=PUB400
DB_USER=your_username
DB_PASSWORD=your_password

โครงสร้างโฟลเดอร์ — แยก route ต่อ 1 endpoint ต่อ 1 ไฟล์ (เพิ่ม endpoint ใหม่ = เพิ่มไฟล์ใหม่ ไม่ต้องแก้ของเดิม)

my-iws-api/
  ├── .env
  ├── server.js        ← จุดรวม ผูก route ทั้งหมดเข้าด้วยกัน
  ├── db.js             ← ฟังก์ชันเชื่อมต่อ ODBC ใช้ร่วมกันทุก endpoint
  └── routes/
        ├── minpop.js    ← endpoint นี้
        └── talinf.js    ← เพิ่มทีหลังตามแบบเดียวกัน
// db.js — ฟังก์ชันเชื่อมต่อกลาง ใช้ซ้ำได้ทุก route
require('dotenv').config();
const odbc = require('odbc');

const connStr =
  `DSN=${process.env.DB_DSN};UID=${process.env.DB_USER};PWD=${process.env.DB_PASSWORD}`;

async function getConnection() {
  return odbc.connect(connStr);
}

module.exports = { getConnection };
// routes/minpop.js — endpoint แทน SCANNER/minpop เดิม
const express = require('express');
const router = express.Router();
const { getConnection } = require('../db');

router.get('/minpop', async (req, res) => {
  const empid = req.query.empid;   // ← ชื่อที่ฝั่งเว็บใช้เรียก (เหมือนที่ IWS ตั้งไว้เดิม)

  if (!empid) {
    return res.status(400).json({ error: 'empid is required' });
  }

  try {
    const connection = await getConnection();

    // query ด้วยชื่อ field จริงในตาราง (CDOPCD) ไม่ใช่ empid
    const result = await connection.query(
      'SELECT CDOPCD, NMOPCD FROM MYLIB.OPERATOR WHERE CDOPCD = ?',
      [empid]
    );
    await connection.close();

    // จุดนี้คือ "IWS mapping layer" ที่จำลองขึ้นเอง —
    // แปลงชื่อ field จริง (CDOPCD/NMOPCD) เป็นชื่อที่เว็บคุ้นเคย (P_EMPNAME/P_ISFOUND)
    if (result.length === 0) {
      return res.json({ P_EMPNAME: "", P_ISFOUND: "0" });
    }
    res.json({
      P_EMPNAME: result[0].NMOPCD,
      P_ISFOUND: "1"
    });

  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

module.exports = router;
// routes/talinf.js — เทมเพลตเปล่า เตรียมไว้สำหรับตอนต่อ endpoint นี้จริง
const express = require('express');
const router = express.Router();
const { getConnection } = require('../db');

router.get('/talinf', async (req, res) => {
  // TODO: แก้ query/field ให้ตรงกับตาราง TALINF จริงตอนพร้อม
  res.status(501).json({ error: 'talinf endpoint not implemented yet' });
});

module.exports = router;
// server.js — จุดเดียวที่ผูก route ทั้งหมดเข้าด้วยกัน
require('dotenv').config();
const express = require('express');
const cors = require('cors');

const minpopRoute = require('./routes/minpop');
const talinfRoute = require('./routes/talinf');   // เพิ่ม route ใหม่ตรงนี้เวลามีเพิ่ม

const app = express();
app.use(cors());

app.use('/api', minpopRoute);
app.use('/api', talinfRoute);

app.listen(3001, () => console.log('API running: http://localhost:3001/api'));
ขยายในอนาคตทำแค่ 3 ขั้นตอน: (1) สร้างไฟล์ใหม่ใน routes/ เช่น routes/talinf.js (2) เขียน query + mapping ในไฟล์นั้น (3) เพิ่ม 1 บรรทัด require + app.use ใน server.js — ไม่ต้องแตะ minpop.js หรือ db.js เลย
5
รันแล้วทดสอบ (เหมือนตอนทดสอบ IWS URL ตรงๆ)
node server.js

เปิด browser ไปที่:

http://localhost:3001/api/minpop?empid=25066

ควรได้ผลลัพธ์ (แม้ query ข้างในใช้ CDOPCD/NMOPCD แต่ response ที่ออกมาหน้าตาเหมือนของเดิมเป๊ะ เพราะผ่าน mapping ใน Step 4 แล้ว):

{ "P_EMPNAME": "I.KIDAKORN", "P_ISFOUND": "1" }
ถ้าได้แบบนี้ แปลว่า endpoint ของคุณทำงานเหมือน IWS เดิมทุกประการแล้ว 🎉
6
ต่อกับหน้าเว็บเดิม — แก้แค่ 1 บรรทัด

ในโปรเจกต์ Next.js เดิม (iws_webAS400_next) เปลี่ยนแค่ค่าใน config:

// เดิม (บริษัท)
NEXT_PUBLIC_API_BASE_URL=/api/web/serveices

// เปลี่ยนเป็น (ฝึกกับ pub400)
NEXT_PUBLIC_API_BASE_URL=http://localhost:3001/api

และถ้า endpoint path ต่างชื่อ (/minpop vs /scanner/minpop) ก็แก้แค่ตรงนั้นใน config เดียวกัน — useMinpopSearch.ts และ component อื่นๆ ไม่ต้องแตะเลย

สรุปย้ำ — ทำไมวิธีนี้ถึง "คล้ายกัน" จริงๆ

  • รูปแบบ URL + query string (?empid=...) เหมือนเดิม
  • รูปแบบ response JSON (P_EMPNAME, P_ISFOUND) เหมือนเดิม เพราะจงใจ mapping ให้ตรง แม้ field จริงในตารางจะชื่อ CDOPCD/NMOPCD ก็ตาม
  • โค้ดฝั่ง React/Next.js (hooks, component, error handling) เหมือนเดิม 100%
  • สิ่งที่ต่างคือ "ใครสร้าง endpoint และใครทำ mapping" — IWS ให้แอดมินตั้งค่าผ่านหน้าจอ / วิธีนี้คุณเขียน mapping เองในโค้ด
🔴 ข้อจำกัดที่ต้องรู้: Node.js API นี้ต้องรันอยู่บนเครื่อง PC ตลอดเวลาที่ใช้งาน (ปิดเครื่อง = endpoint หายไปด้วย) ต่างจาก IWS ที่รันอยู่บน server ตลอดเวลา — เหมาะกับ "ฝึกส่วนตัว" เท่านั้น ไม่ใช่ระบบที่จะให้คนอื่นใช้งานจริง
