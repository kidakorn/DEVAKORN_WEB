iws_webAS400_next — Roadmap ต่อยอด

github.com/kidakorn/iws_webAS400_next

สรุปสถานะปัจจุบันของโปรเจกต์จริงที่ทำร่วมกับบริษัท (IWS + Next.js) และแผนพัฒนาต่อจากนี้ เรียงตามลำดับที่ควรทำก่อน-หลัง

สถานะตอนนี้

✅ ทำสำเร็จแล้ว

  • Setup next.config.ts เป็น Static Export ถูกต้อง (output: 'export', basePath: '/kidakorn', trailingSlash: true)
  • โครงสร้างโปรเจกต์ clean code: app/, components/, hooks/, lib/, types/ แยกชัดเจน
  • ดึงข้อมูลจริงจาก IWS สำเร็จ — หน้า minpop (Operator Code Search) ใช้งานได้จริง
  • มี validation, error handling, SweetAlert2 popup แจ้งเตือนครบ
  • Deploy ไปโฟลเดอร์แชร์บริษัทสำเร็จ ผ่าน as400webappdev/kidakorn
  • บังคับเช็ก NEXT_PUBLIC_API_BASE_URL ตอน build กันลืมตั้งค่า
  • หน้า talinf — โค้ดตรวจสอบแล้วครบ: client.ts เป็นตัวกลางที่ถูกต้อง, lib/api/minpop.ts และ lib/api/talinf.ts ใช้ apiRequest() pattern เดียวกัน 1:1, useTalinfSearch.ts เขียนตาม pattern เดียวกับ minpop ครบ

🕐 รอทดสอบพรุ่งนี้ที่บริษัท (ไม่ใช่ของค้างในโค้ดแล้ว)

  • ทดสอบหน้า talinf กับ lot number จริงผ่าน URL production (as400webappdev/kidakorn/talinf/) — โค้ดพร้อมแล้ว รอแค่ยืนยันกับข้อมูลจริง
  • ถ้าผ่าน → ปิดจ็อบ talinf ได้อย่างเป็นทางการ ไม่ต้องกลับมาแก้โค้ดส่วนนี้อีก

🚧 ของค้างอื่นที่ยังเหลือจริง

  • console.log debug ที่เคยใส่ไว้ชั่วคราวใน useMinpopSearch.ts — เช็คว่าลบออกแล้วหรือยัง

แผนพัฒนาต่อ

1
ปิดจ็อบที่เหลือ (ทำก่อนสุด)
เกือบจบแล้ว เหลือแค่ยืนยันของจริง
  • ทดสอบ talinf กับ lot number จริงที่บริษัทพรุ่งนี้ — โค้ดพร้อมแล้ว 100% รอแค่ยืนยันผล
  • ลบ console.log debug ที่เหลือใน useMinpopSearch.ts ออกให้หมด
  • ทดสอบทั้ง 2 หน้า (minpop, talinf) พร้อมกันบน production URL จริงอีกรอบให้ชัวร์
  • พิจารณา refactor เล็กน้อย: ทั้ง 2 endpoint มี logic ซ้ำกันเยอะมาก (validate → fetch → handle state → mock) อาจดึงเป็น generic hook กลาง เช่น useSearchResource() ให้ endpoint ใหม่ในอนาคตเขียนโค้ดซ้ำน้อยลง
2
ปรับปรุงประสบการณ์ใช้งาน (UX)
ฟีเจอร์เล็กๆ ที่เพิ่มคุณภาพงานได้เร็ว
  • Loading skeleton แทนข้อความ "Loading..." ธรรมดา (มี shadcn/ui อยู่แล้ว ใช้ Skeleton component ได้เลย)
  • เก็บประวัติค้นหาล่าสุด (recent search) ไว้ใน localStorage ให้กดซ้ำได้เร็ว
  • QR code แบบ print-friendly สำหรับพิมพ์เป็นบัตร/ป้าย
  • Responsive check บนมือถือให้ครบ (ปุ่ม, ตาราง, popup)
3
ขยาย endpoint ใหม่
ต้องคุยกับแอดมิน/ทีม RPG
  • รวมข้อมูล minpop + talinf เป็นหน้าเดียวถ้าใช้งานคู่กันบ่อย (tab หรือ combined view)
  • ขอ endpoint แบบ list/pagination ถ้าต้องการดูข้อมูลหลาย record พร้อมกัน
  • ใช้ pattern spec ที่เคยร่างไว้ก่อนหน้าตอนขอแอดมิน deploy service ใหม่
4
Production hardening
ก่อนให้คนอื่นในบริษัทใช้งานจริงเป็นวงกว้าง
  • แยก .env.production / .env.local ให้ชัดเจน ไม่ปนกัน
  • คุยกับแอดมินเรื่องจำกัด CORS header ให้เจาะจงโดเมนจริง (ไม่ใช้ *)
  • เพิ่ม error boundary/monitoring เบื้องต้น (เช่น log error ไปที่ไหนสักที่เวลาเกิด 500/network error)
  • เขียน README/เอกสารส่งมอบให้คนอื่นในทีมเข้าใจ pattern config → api → hooks → page
